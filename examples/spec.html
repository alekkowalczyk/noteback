RealtimeSync Service — Technical Spec

Draft v0.3 · generated for review

Overview

RealtimeSync keeps client documents in sync with the server in near real time. Clients open a WebSocket connection and receive change events as other users edit the same document. The server guarantees that every message is delivered at least once, and clients are responsible for de-duplicating events they have already seen.

Architecture

Incoming edits are written to a single Redis instance that acts as the central message broker. Each connected client subscribes to a channel named after the document id. When an edit arrives, the broker fans it out to every subscriber on that channel synchronously before acknowledging the writer.

Data Model

Each user has a single workspace, and every document belongs to exactly one workspace. Permissions are stored as a flat list of user ids on the document itself, which is checked on every incoming edit.

Scaling

To handle more load we will run multiple server instances behind a load balancer. Because all state lives in Redis, any instance can serve any client, and we can add instances without coordination. We expect this to scale linearly to roughly one million concurrent connections.
