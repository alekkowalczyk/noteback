Replace Noteback's two divergent persistence models with one timeline of full-document versions — each a complete, re-openable canvas, not a fragment.

Two of your instincts started this: that the no-hash degrade path could fall back to a minted id instead of giving up, and that the content-hash machinery — which extension mode doesn't use at all — would help there too. Both point at one root cause: identity is derived at runtime, and the two modes derive it differently. This design makes identity explicit, and the history model identical across modes.

00

The problem

Two stories, one of them blank

There are two unrelated persistence stories today, and only one of them has any history at all:

 ExtensionEmbedded
IdentityFull URL (location.href)Content hash + normalized URL
Backendchrome.storage.localin-file JSON + localStorage
Content hash?NoYes
History?None at allYes — but stored as padded fragments

Embedded "earlier feedback" stores bits and pieces: padded "sections" around each selection, re-highlighted across text nodes with whitespace-loose matching, then trimmed to a byte cap. That subsystem (snapshot.js, ~311 lines) is where several shipped bugs lived — empty captures, first-mark-only unions, a paint-before-persist trap. Meanwhile the extension, where the author's real iterate-and-re-comment loop happens, has no history whatsoever.

01

Mental model

One timeline of versions

Collapse "current comments" and "earlier feedback" into a single concept: a document owns an ordered list of versions, and each version is a complete, openable document with its comments attached.

The core shift

A history entry becomes the same kind of thing as the live document — a whole doc you open and read in context — not a clipped fragment that must be reconstructed. That equivalence is what removes the complexity.

02

Identity

Baked when we own the HTML; stored by URL when we don't

You flagged the asymmetry exactly: the extension doesn't control the page's HTML, so it can't bake anything into it. So identity is one concept with two acquisition paths:

  • wrap / "Save as canvas" → a stable doc-id is baked into the document (e.g. data-noteback-doc-id on the root). It travels with the file, survives renames and moves, and wrap preserves it on re-export.
  • extension on a page it didn't author → doc-id is looked up (or minted) in a chrome.storage URL → doc-id map — today's attach map, but pointing at a stable id instead of being the identity.
  • A baked id always wins over a URL lookup. Land on a wrapped canvas → read the baked id and chain correctly. Save a bare page as a canvas → that's where a URL-keyed page gets a baked id minted into the output.

The per-version key is the content hash when available, and a minted gen-id when the content is too short to hash — your "nonce," now first-class instead of a fallback that gives up.

✓ Decided

Identity: baked doc-id preserved across re-wrap; URL→id map in the extension when the HTML isn't ours; baked wins.
Extension snapshot scope: on by default for file:// / localhost / wrapped canvases; opt-in per site for arbitrary http(s) pages — riding the existing origin-policy / nb:settings per-origin mechanism, so no new settings system.

03

Experience

One sidebar, every stage of the doc's life

It's a single panel that adapts to where you are — from a blank first read, to an active draft with a deep history, to peeking an old version, to the edge cases where history can't (or shouldn't) exist.

A · The working state — current draft + version timeline

The everyday view. Your live comments are pinned at the top; every earlier version that received feedback is a node on the timeline below, newest first. Each past row offers three actions.

Rule · keep the panel calm

The Versions group scales with how much history there is: hidden at 0 earlier versions, shown inline at exactly 1, and at 2 or more the most-recent earlier version stays inline while the rest tuck under a "+N older versions" disclosure. So the last round of feedback is always zero-click, deeper history is one click, and the current draft is never buried.

B · Reading a past version — peek & "Back to current"

Click a row and it expands in place: the snapshot's comments list inline, and a banner pins to the top so you always have a one-click way home. open promotes the same version to a full canvas in a new tab — scroll it, save it, copy from it, exactly like the live doc.

Wording · locked

The "return to live" control reads "Back to current" (see §7).

C · The edge states

The same panel handles the cases the mechanism implies — a brand-new doc, storage that can't help, a site you haven't opted in, and a version old enough to have been pruned.

First read — nothing yet

No comments, no history. The panel invites the first action and hides the Versions list until there's something in it.

History unavailable — degrades gracefully

Storage is blocked, or the content is too short to fingerprint. Comments still save (in-file / per-URL); only the timeline steps aside.

A site you didn't author — opt-in

In the extension, arbitrary http(s) pages don't snapshot until you turn it on, per site (rides nb:settings). Review contexts are on by default.

An old version — snapshot pruned

Beyond the retention window the heavy snapshot is evicted but the feedback is kept. open dims; copy feedback still works.

04

Mechanism

How it works

aOne engine, two backends

draft-history-core.js is already storage-agnostic (it takes an injected store). Embedded keeps the localStorage kv; the extension gets a thin chrome.storage kv shim feeding the same engine. The mode divergence disappears — one history implementation.

bSnapshot trigger: first comment of a version

When a version's comment count goes 0 → 1, capture the clean full-doc HTML: clone the document, strip [data-noteback-ui], unwrap <mark>, drop the runtime <script> and state block — the routine that already exists as docContentHtml() / rebuildCleanHtml(). gzip and store. A version's content is fixed, so it's captured once; later comments only update the list.

cRendering history = the live painter

Because we store the clean doc plus comment anchors (not painted marks), showing a historical version just runs the normal highlight painter against the snapshot. No cross-node whitespace-loose re-highlighter; no padded-section reconstruction.

dKey spaces

KeyHolds
nb:doc:<doc-id>Lineage: title, ordered version list (key, createdAt, commentCount, hasSnapshot)
nb:ver:<version>One version: doc-id, contentHash, comments[], gzipped clean snapshot, timestamps
nb:url:<url>→ doc-id (extension only; used when no baked id is present)
05

Payoff

What it retires

✗ Retires
  • Most of snapshot.js (~311 lines): extractSections, padContext, trimToCap, section union/dedupe.
  • The cross-node, whitespace-loose nbHistHighlight re-highlighter — replaced by the normal painter.
  • The "paint before persist" bug class — structurally impossible now; snapshots no longer read painted marks from the DOM.
  • The embedded-vs-extension behavioral gap — one engine, one behavior.
06

Risks

What to watch

⚠ Storage

Whole-doc > fragments. Bounded by gzip + keep last ~10 commented versions per doc + a global byte cap (most of which pruneLineage already does). The embedded file:// path shares one localStorage bucket across all local canvases, so the cap matters most there; the extension's chrome.storage is per-extension, no shared-bucket pressure. Recommendation: defer unlimitedStorage — the per-site opt-in keeps arbitrary pages out by default; add it only if real docs blow the quota.

⚠ Re-wrap continuity

The one genuinely tricky bit. For v1 → v2 history to chain across separately-shared files, wrap must preserve the baked id. Proposal: wrap input.html -o out.canvas.html reads the id from out if it already exists, plus a --id <id> escape hatch. Worst case it mints a new lineage — graceful, same as today.

⚠ Fidelity

Snapshots are self-contained for wrapped canvases (inline styles). An arbitrary extension page's external CSS / remote images won't load on checkout — the same limitation as today's popup, just full-structure instead of fragments.

07

Locked

Decisions

✓ Locked for the plan
  • Checkout depth — full: inline peek + open as a live canvas tab + copy feedback.
  • Re-wrap continuity — yes: wrap preserves the baked doc-id (reuse from the -o target, plus a --id flag).
  • "Return to live" wording — "Back to current".
  • Sequencing — both modes at once: one cutover to the shared engine + both kv backends.
  • Collapsed history — show the most-recent earlier version inline; collapse the rest under "+N older versions".
  • Retention — reuse existing limits (≈5 full snapshots · 15 metadata · 90-day TTL · ~3 MB cap).
  • Migration — clean break: no migration code. Embedded current comments still load from the in-file block; extension per-URL comments reset on upgrade.
  • Identity — baked doc-id (wins) + extension URL→id map; unlimitedStorage deferred.

The implementation plan follows from these. Change anything here and the plan changes with it.
